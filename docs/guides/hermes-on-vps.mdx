---

This guide takes you from two empty Hetzner VPS boxes to a long-running Hermes Agent that calls Anthropic, GitHub, Slack, and Telegram without ever holding a real credential on the agent host. Allow about 60–90 minutes; budget around €10/month for the two boxes.

By the end, you'll have:

1. **Agent Vault running on one VPS**, holding real Anthropic, GitHub, Slack, and Telegram credentials encrypted at rest behind a master password.
2. **A [Hermes Agent](https://github.com/NousResearch/hermes-agent) running on a second VPS**, reachable via a Telegram bot, with every outbound API call brokered through Agent Vault — and revocable from the UI in one click.

## Prerequisites

- A [Hetzner Cloud](https://www.hetzner.com/cloud) account.
- An Anthropic API key (`sk-ant-...`), a GitHub PAT with `repo` scope, and a Slack bot token (`xoxb-...`) with `chat:write`, `channels:read`, and `groups:read`. Invite the Slack bot to a channel (e.g. `#ai-updates`).
- A Telegram account on your phone.

<Note>
This guide installs Agent Vault as a native binary via the install script — the path documented in [Self-hosting locally](/self-hosting/local). For containers, see [Docker self-hosting](/self-hosting/docker).
</Note>

## Architecture

```text
                Your laptop (browser + phone for Telegram)
                          │
                          │ SSH (locked to your laptop IP) + SSH tunnel for AV UI
                          │
    Public Internet ──────┼──────────────────────────────────────
                          │                                     │
                ┌─────────▼──────────┐               ┌──────────▼──────────┐
                │ av-broker (10.0.0.2)│               │ hermes-vps (10.0.0.3)│
                │ Agent Vault server  │◀──HTTPS_PROXY─┤ Hermes + gateway     │
                │  :14321 control     │   (private)   │ Placeholders only    │
                │  :14322 MITM        │               │                      │
                └─────────┬───────────┘               └─────────────────────┘
                          │
                          │ Real credentials injected here
                          ▼
              api.anthropic.com   api.github.com
              slack.com           api.telegram.org
```

Only `av-broker` holds real credentials, encrypted at rest behind your master password. `hermes-vps` holds placeholder strings (`__anthropic_api_key__`, etc.) and one revocable [agent token](/agents/overview).

## 1. Provision the VPS boxes

<Steps>

<Step title="Create a Hetzner project">

In the [Hetzner Cloud Console](https://console.hetzner.cloud), click **+ New Project**. Name it (e.g. `agent-vault-lab`) and click in. Everything that follows happens inside this project.

</Step>

<Step title="Generate an SSH key on your laptop">

```bash
ssh-keygen -t ed25519 -f ~/.ssh/agent-vault-lab -C "agent-vault-lab"
```

Press Enter twice at the passphrase prompts to leave it empty. Every `ssh` command in this guide uses `-i ~/.ssh/agent-vault-lab` to pin the connection to this specific key, so `ssh-agent` doesn't try other keys and trip Hetzner's attempt cap.

</Step>

<Step title="Upload the public key to Hetzner">

In the project sidebar: **Security → SSH Keys → Add SSH Key.** Paste the contents of `~/.ssh/agent-vault-lab.pub`. Name it (e.g. `lab`).

**Tick "Make default for this project"** before saving. This auto-attaches the key to every server you create in the project, which is what lets you skip the SSH-key picker in step 5.

</Step>

<Step title="Create the private network">

**Networks → Create Network.** Set:

- **Name:** `av-net`
- **IP range:** `10.0.0.0/24`
- **Network zone:** any (e.g. `eu-central`)

Remember the network zone — you'll match it when creating servers.

</Step>

<Step title="Create av-broker">

**Servers → + Add Server.** Walk the form top to bottom:

| Panel | Setting |
| --- | --- |
| **Location** | Any datacenter in the network zone you picked above (e.g. `nbg1`) |
| **Image** | Ubuntu 24.04 |
| **Type** | `CX23` (or whatever the cheapest x86 tier currently visible is — Hetzner retires tiers, so trust the UI's availability) |
| **Networking** | Expand **Private networks** and tick `av-net`. After it's attached, click the IP field and set it to `10.0.0.2`. |
| **SSH keys** | The key you uploaded should already be ticked (because you ticked "Make default"). If it isn't, tick it manually. |
| **Name** | `av-broker` |

Click **Create & Buy now**. After ~30 seconds the server's **Status** flips to **Running**. Copy its **public IPv4** from the server's detail page — you need it in step 7.

</Step>

<Step title="Create hermes-vps">

Same form, same answers — same location, image, type, SSH key. Two differences:

- **Networking → Private network IP:** `10.0.0.3`
- **Name:** `hermes-vps`

Click **Create & Buy now**. Copy this server's public IPv4 too.

</Step>

<Step title="Verify private connectivity">

From your laptop:

```bash
ssh -i ~/.ssh/agent-vault-lab root@<av-broker-public-ip>
ping -c 3 10.0.0.3
```

Three replies confirms `av-broker` and `hermes-vps` can see each other across the private subnet. Type `exit` to disconnect — you'll come back to install Agent Vault.

</Step>

</Steps>

## 2. Restrict SSH to your laptop

<Steps>

<Step title="Find your laptop's public IPv4">

```bash
curl -4 ifconfig.me; echo
```

The `-4` flag forces IPv4 — without it `curl` often returns IPv6.

</Step>

<Step title="Create and attach the firewall">

**Firewalls → Create Firewall**, name `ssh-lockdown`. Add one inbound rule:

- **Protocol:** TCP
- **Port:** `22`
- **Source IPs:** the IPv4 from the previous step followed by `/32` (e.g. `203.0.113.42/32` — the `/32` means "just this one address")

Leave outbound rules empty (Hetzner allows all outbound by default).

After saving, scroll down to **Apply To → Servers** and tick both `av-broker` and `hermes-vps`.

Test from a different device (your phone on cellular): SSH should time out. From your laptop it still works.

</Step>

</Steps>

<Warning>
If your laptop's public IP changes (cafe networks, VPNs), update the firewall rule's source IP.
</Warning>

## 3. Install Agent Vault on av-broker

<Steps>

<Step title="SSH into av-broker">

From your laptop:

```bash
ssh -i ~/.ssh/agent-vault-lab root@<av-broker-public-ip>
```

Every command in this phase runs on `av-broker` until you exit.

</Step>

<Step title="Install Agent Vault">

```bash
curl --proto '=https' --proto-redir '=https' --tlsv1.2 -fsSL https://get.agent-vault.dev | sh
```

The script detects your architecture, verifies the signed release, and drops the binary at `/usr/local/bin/agent-vault`.

</Step>

<Step title="Install tmux">

```bash
apt update && apt install -y tmux
```

`tmux` is a terminal multiplexer. You'll start Agent Vault inside a `tmux` session in the next step — that way the server keeps running on `av-broker` after you disconnect from SSH, instead of being killed when your shell exits. Without it, the server dies the moment you close your laptop or your SSH connection drops.

</Step>

<Step title="Launch the server inside tmux">

```bash
tmux new -s av
agent-vault server --host 10.0.0.2 --port 14321 --mitm-port 14322
```

`--host 10.0.0.2` binds both listeners to the private NIC — Agent Vault is unreachable from the public internet.

First-run prompts: master password (twice), admin email, admin password. Keep the master password in your head and your password manager only.

Detach with `Ctrl-B` then `D`. You'll see `[detached from session av]` and drop back to your normal shell. The server keeps running inside the tmux session.

</Step>

<Step title="Verify the server">

```bash
curl -sI http://10.0.0.2:14321/health
```

Expect `HTTP/1.1 200 OK`.

</Step>

</Steps>

<Note>
On reboot, tmux dies and so does the server. SSH back in, `tmux new -s av`, restart with the same command, re-enter the master password. For zero-touch restart, set [`AGENT_VAULT_MASTER_PASSWORD`](/self-hosting/environment-variables) — but the password then lives in an env file.
</Note>

## 4. Configure the vault

<Steps>

<Step title="Open an SSH tunnel">

In a **new terminal on your laptop** (don't reuse the one you used to install Agent Vault):

```bash
ssh -i ~/.ssh/agent-vault-lab -L 14321:10.0.0.2:14321 root@<av-broker-public-ip> -N
```

The terminal hangs silently — that's correct. **Leave it running for the rest of the guide.** Closing it stops the tunnel and the AV UI goes dark.

Open `http://localhost:14321` in your laptop's browser and sign in with the admin email and password you set during install.

</Step>

<Step title="Understand vaults, credentials, and services">

You'll create one of each. Briefly:

- A [**vault**](/learn/vaults) is a named isolation boundary. We'll create `prod`.
- A [**credential**](/learn/credentials) is one secret value stored under a name (e.g. `ANTHROPIC_API_KEY = sk-ant-...`). Encrypted at rest.
- A [**service**](/learn/services) is the wiring for one upstream host: host pattern + auth mode + substitution rules that reference credentials by name.

Credentials store values. Services define wiring. At request time, Agent Vault matches the service by host, looks up the named credential, and rewrites the wire bytes before forwarding.

</Step>

<Step title="Create the prod vault">

**Vaults → New vault → `prod` → Create.**

</Step>

<Step title="Add four credentials">

In the `prod` vault, **Credentials tab → Add credential.** Add four rows:

| Key | Value |
| --- | --- |
| `ANTHROPIC_API_KEY` | Your real Anthropic key (`sk-ant-...`) |
| `GITHUB_TOKEN` | Your real GitHub PAT |
| `SLACK_BOT_TOKEN` | Your real Slack bot token (`xoxb-...`) |
| `TELEGRAM_BOT_TOKEN` | Placeholder for now — you create the real bot in the next section |

</Step>

<Step title="Add four services">

A substitution rule tells Agent Vault: "in requests to this host, scan the **surface** (header / path / query) for the **placeholder** string and replace it with the **credential**'s value before forwarding." Hermes will emit the placeholders on the wire; the real keys never leave `av-broker`.

**Services tab → + Add service.** For each row below: fill **Name** and **Host**, set **Authentication** to **Passthrough**, then scroll to **URL Substitutions** and click **+ Add substitution**. Paste the placeholder, tick exactly one **Surface** box (untick the others), select the credential. Save.

| Name | Host | Placeholder | Surface | Credential |
| --- | --- | --- | --- | --- |
| `anthropic-brain` | `api.anthropic.com` | `__anthropic_api_key__` | `header` | `ANTHROPIC_API_KEY` |
| `github` | `api.github.com` | `__github_token__` | `header` | `GITHUB_TOKEN` |
| `slack` | `slack.com/*` | `__slack_bot_token__` | `header` | `SLACK_BOT_TOKEN` |
| `telegram` | `api.telegram.org` | `__telegram_bot_token__` | **`path`** | `TELEGRAM_BOT_TOKEN` |

Three services use `header` (the upstream reads the credential from an HTTP header). Telegram uses `path` because its Bot API encodes the token inline in the URL (`/bot<TOKEN>/sendMessage`).

</Step>

</Steps>

<Warning>
**Placeholder strings are exact-match.** `__anthropic_api_key` (one trailing underscore) will not match `__anthropic_api_key__` (two). If you later see a `401: invalid x-api-key` from an upstream, open **Logs** in the AV UI, click the failing request, and check the **Credential keys** field. Empty means the substitution didn't fire — fix the string.
</Warning>

## 5. Create the Telegram bot

<Steps>

<Step title="Mint the bot via BotFather">

In Telegram, message `@BotFather`. Send `/newbot`, give it a display name, then a username ending in `bot`. BotFather returns a token: `123456789:ABC-DEF...`.

</Step>

<Step title="Paste the real token into Agent Vault">

In the AV UI: `prod` → **Credentials** → `TELEGRAM_BOT_TOKEN` → edit → paste the BotFather token → save. The token never lives on `hermes-vps`.

</Step>

<Step title="Get your numeric Telegram user ID">

Message `@userinfobot`, send `/start`. Copy the numeric `Id:` — you'll need it when configuring the Hermes gateway to lock the bot to your account.

</Step>

</Steps>

## 6. Connect Hermes

<Steps>

<Step title="SSH into hermes-vps and install the Agent Vault CLI">

In a **fresh terminal on your laptop** (leave the SSH tunnel to `av-broker` running):

```bash
ssh -i ~/.ssh/agent-vault-lab root@<hermes-vps-public-ip>
curl --proto '=https' --proto-redir '=https' --tlsv1.2 -fsSL https://get.agent-vault.dev | sh
```

Every command from here through starting the gateway runs on `hermes-vps`. On this box the AV CLI is a client wrapper — no server runs here.

</Step>

<Step title="Install Hermes and walk the setup wizard">

```bash
curl -fsSL https://raw.githubusercontent.com/NousResearch/hermes-agent/main/scripts/install.sh | bash
```

The install script drops Hermes at `/usr/local/lib/hermes-agent/` and then launches an interactive four-question setup wizard. Answer:

1. **How would you like to set up Hermes?** → **Quick setup**.

![Hermes first-launch setup: Quick setup vs Full setup](/images/hermes-on-vps/hermes-on-vps-wizard-setup.png)

2. **Select provider:** scroll to the bottom and choose **Leave unchanged**. The next step writes placeholder lines to `~/.hermes/.env`; you don't want the wizard baking in any provider config that would compete with that.

![Hermes provider list with Leave unchanged selected at the bottom](/images/hermes-on-vps/hermes-on-vps-wizard-provider.png)

3. **Select terminal backend:** **Keep current (local)**. Hermes runs directly on `hermes-vps`; no sandbox needed.

![Hermes terminal backend list with Keep current local selected](/images/hermes-on-vps/hermes-on-vps-wizard-terminal.png)

4. **Connect a messaging platform?** **Skip — set up later with `hermes setup gateway`**. You'll configure Telegram in a later section using the gateway wizard.

![Hermes messaging platform prompt with Skip selected](/images/hermes-on-vps/hermes-on-vps-wizard-messaging.png)

The wizard exits and you're back at the shell prompt.

</Step>

<Step title="Write placeholder credential lines into Hermes's .env">

```bash
cat > ~/.hermes/.env <<'EOF'
ANTHROPIC_API_KEY=__anthropic_api_key__
GITHUB_TOKEN=__github_token__
SLACK_BOT_TOKEN=__slack_bot_token__
TELEGRAM_BOT_TOKEN=__telegram_bot_token__
EOF
```

These placeholders match the substitution rules you created earlier. Hermes pushes them onto the wire; Agent Vault swaps in the real values at the proxy boundary.

</Step>

<Step title="Mint an agent token (back to your laptop's AV UI)">

Switch to the AV UI tab in your laptop browser. Sidebar: **All Agents → Add agent.**

- **Agent name:** `hermes-vps`
- **Vault access:** click **+ Add vault**, pick `prod`, leave role at `proxy`
- Click **Add**

The modal flips to the **Connect Your Agent** view with a copyable snippet of three `export` lines — broker address and agent token pre-filled, vault name as a placeholder. Click the inline **Copy** button. You'll paste it in the next step.

</Step>

<Step title="Export the three AV env vars">

Back on the `hermes-vps` terminal, paste the snippet. Replace `<VAULT_NAME>` with `prod`:

```bash
export AGENT_VAULT_ADDR="http://10.0.0.2:14321"
export AGENT_VAULT_TOKEN="<long-token-from-UI>"
export AGENT_VAULT_VAULT="prod"
```

Each one answers a different question: **which broker** (`ADDR`), **who is calling** (`TOKEN`, revocable from the UI), and **which set of credentials and services applies** (`VAULT`).

</Step>

<Step title="Sanity-check the proxy path">

```bash
agent-vault run -- hermes
```

Look for `agent-vault: routing HTTP/HTTPS through MITM proxy (10.0.0.2:14322)` near the top of the output. Send a prompt to confirm Claude replies through the broker, then `Ctrl-C` to quit.

</Step>

</Steps>

## 7. Configure the Hermes gateway

The gateway is Hermes's always-on messaging daemon. Configure it via a wizard wrapped in `agent-vault run` so its environment has the broker config.

<Steps>

<Step title="Launch the wizard">

```bash
agent-vault run -- hermes gateway setup
```

</Step>

<Step title="Walk the wizard">

Telegram already shows as `configured` in the platform list because the placeholder line you wrote to `~/.hermes/.env` earlier is enough for the wizard to consider it set up. You'll skip reconfiguring the token and just add the user allowlist.

1. **Select a platform to configure** → **Telegram** (shown as `(configured)`).

![Hermes gateway wizard platform list with Telegram shown as configured](/images/hermes-on-vps/hermes-on-vps-gateway-platforms.png)

2. **Reconfigure Telegram?** → `N`.

3. **Add allowed users now?** → `Y`.

4. **Allowed user IDs (comma-separated)** → your numeric Telegram user ID from `@userinfobot`. This locks the bot to your account.

5. **Select a platform to configure** (you're back at the platform list) → scroll to the bottom and select **Done**.

![Hermes gateway wizard platform list with Done at the bottom](/images/hermes-on-vps/hermes-on-vps-gateway-done.png)

6. **Install the gateway as a systemd service?** → `Y`.

7. **Choose how the gateway should run** → **System service** (the second option). System service writes the unit to `/etc/systemd/system/` and starts on boot natively — no `loginctl enable-linger` needed.

![Hermes gateway wizard service-type prompt with System service selected](/images/hermes-on-vps/hermes-on-vps-gateway-service-type.png)

8. **Run as which user?** → `root`.

9. **Start the service now?** → **`N`**. The unit has no Agent Vault env yet; the next section wires it before starting.

![Hermes gateway wizard start-now prompt with No selected](/images/hermes-on-vps/hermes-on-vps-gateway-start.png)

The installer writes `/etc/systemd/system/hermes-gateway.service` and enables it on boot.

</Step>

</Steps>

## 8. Start the gateway daemon

The unit needs Agent Vault's environment wired explicitly (systemd doesn't inherit it from your shell). Bump one rate-limit knob in the UI, then run three pastes on `hermes-vps`.

<Steps>

<Step title="Raise the AUTH rate limit">

In the AV UI tab on your laptop: click your email (top right) → **Manage Instance** → scroll to **Rate Limits**. In the **AUTH** row, change `Max` from `10` to `10000`. Save.

The default 10-per-5-minute auth limit locks a long-running daemon out in seconds — the gateway's `httpx` client opens a fresh CONNECT every retry. See [Instance settings](/guides/instance-settings) for the full knob reference.

<Note>
The auth tier throttles brute force at the login endpoint. Agent Vault is bound to `10.0.0.2` and only `hermes-vps` can reach it, so the per-IP gate isn't load-bearing here. Keep it tight if you ever bind to `0.0.0.0`.
</Note>

</Step>

<Step title="Paste 1 — write /root/.hermes/gateway.env">

In the SSH shell on `hermes-vps` where `$AGENT_VAULT_TOKEN` is still set:

```bash
if [ -z "$AGENT_VAULT_TOKEN" ]; then
  echo "✗ AGENT_VAULT_TOKEN not set — re-export from the Connect Hermes section"
else
  cat > /root/.hermes/gateway.env <<EOF
AGENT_VAULT_ADDR=http://10.0.0.2:14321
AGENT_VAULT_TOKEN=$AGENT_VAULT_TOKEN
AGENT_VAULT_VAULT=prod
HTTPS_PROXY=https://$AGENT_VAULT_TOKEN@10.0.0.2:14322
HTTP_PROXY=https://$AGENT_VAULT_TOKEN@10.0.0.2:14322
REQUESTS_CA_BUNDLE=/root/.agent-vault/mitm-ca.pem
SSL_CERT_FILE=/root/.agent-vault/mitm-ca.pem
NODE_EXTRA_CA_CERTS=/root/.agent-vault/mitm-ca.pem
CURL_CA_BUNDLE=/root/.agent-vault/mitm-ca.pem
GIT_SSL_CAINFO=/root/.agent-vault/mitm-ca.pem
DENO_CERT=/root/.agent-vault/mitm-ca.pem
NODE_USE_ENV_PROXY=1
EOF
  chmod 600 /root/.hermes/gateway.env
  ls -la /root/.hermes/gateway.env
fi
```

Expect `-rw------- 1 root root <~600–700> ... gateway.env`. If you see the warning instead, re-run the `export AGENT_VAULT_TOKEN=...` line from the previous section.

</Step>

<Step title="Paste 2 — systemd drop-in">

```bash
mkdir -p /etc/systemd/system/hermes-gateway.service.d
cat > /etc/systemd/system/hermes-gateway.service.d/override.conf <<'EOF'
[Service]
EnvironmentFile=/root/.hermes/gateway.env
EOF
systemctl daemon-reload
cat /etc/systemd/system/hermes-gateway.service.d/override.conf
```

Expect two lines from the trailing `cat`: `[Service]` and `EnvironmentFile=/root/.hermes/gateway.env`.

</Step>

<Step title="Paste 3 — start and verify">

```bash
systemctl start hermes-gateway
sleep 3
systemctl status hermes-gateway --no-pager | head -20
ss -tnp state established | grep -E 'python|14322' || echo "(none yet — wait a few seconds and re-run)"
```

`systemctl status` should show `Active: active (running)`, a `Drop-In:` line, and `Main PID: ... (python)`. `ss` should show one or two rows pointing at `10.0.0.2:14322` — that's the Telegram long-poll going through Agent Vault.

</Step>

</Steps>

<Note>
A repeating `[Slack] SLACK_APP_TOKEN not set` warning in `journalctl -u hermes-gateway` is expected. The gateway sees `SLACK_BOT_TOKEN` in `~/.hermes/.env` and tries to register Slack **inbound**, which needs a separate `xapp-` app token. We don't use Slack inbound — the gateway logs the error, skips it, and keeps running. Telegram inbound and Slack outbound both still work.
</Note>

### If the service fails to start

```bash
journalctl -u hermes-gateway -n 50 --no-pager
head -5 /root/.hermes/gateway.env
```

The token line is the usual culprit. If `$AGENT_VAULT_TOKEN` was empty when you ran paste 1, the `if` guard skipped the heredoc. Re-export, redo paste 1, then `systemctl daemon-reload && systemctl restart hermes-gateway`.

## 9. Lock down egress (optional)

Restrict `hermes-vps` to talking only to `av-broker` so a compromised Hermes has nowhere to exfiltrate to.

<Steps>

<Step title="Create and attach the egress firewall">

**Firewalls → Create Firewall**, name `hermes-egress`. One outbound rule: TCP, destination `10.0.0.0/24`, any port.

In **Apply To → Servers**, select **only** `hermes-vps`. Do not attach this to `av-broker` — that box needs public internet to reach upstream APIs.

</Step>

<Step title="Verify">

From `hermes-vps`:

```bash
curl --max-time 5 https://api.anthropic.com/v1/models
```

That should time out — the firewall blocks direct egress. Routed through the proxy (`10.0.0.2:14322`, inside the allowed range), the same call succeeds.

</Step>

</Steps>

## 10. Try it end-to-end

Send a prompt to your Telegram bot:

> Summarize the last five commits in `<owner>/<repo>` and post the summary to `#ai-updates` on Slack.

In a separate SSH session on `hermes-vps`:

```bash
journalctl -u hermes-gateway -f
```

Watch the GitHub call, the Anthropic completion, and the Slack post fly past. All four calls route through `10.0.0.2:14322` — Agent Vault substitutes placeholders for real credentials at the wire boundary. No real keys ever existed on `hermes-vps`.

Open **Logs** in the AV UI to confirm. Each request shows the matched service and which credential placeholder got substituted.

## 11. Revoke and recover

<Steps>

<Step title="Revoke the agent">

In the AV UI: **Agents → `hermes-vps` → Rotate** (or **Delete**).

</Step>

<Step title="Send another Telegram message">

The next outbound call returns `401` from the broker. The placeholder strings on `hermes-vps` are inert without a valid agent token.

</Step>

<Step title="Re-issue to recover">

Copy the rotated token, update `AGENT_VAULT_TOKEN` in `/root/.hermes/gateway.env`, then `systemctl restart hermes-gateway`. Hermes replies normally. See [Agents → Rotating an agent token](/agents/overview#rotating-an-agent-token).

</Step>

</Steps>

## Cleanup

1. **Hetzner Cloud Console → Servers** — delete `av-broker` and `hermes-vps`.
2. **Networks → `av-net`** — delete.
3. **Firewalls** — delete `ssh-lockdown` and `hermes-egress`.
4. Rotate or revoke the real upstream API tokens you uploaded (they remain valid in their home services until you do).
